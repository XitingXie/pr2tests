# AppTest: AI-Powered Test Generation & Execution Framework

## Project Overview

AppTest is a CLI tool that integrates into CI pipelines to automatically generate and execute test cases for Android apps. It reads PR diffs, understands what changed using LLM reasoning, generates natural language test cases, and executes them on real devices or emulators.

The tool is designed as a CI plugin that runs in the customer's infrastructure. Source code never leaves their environment. Only relevant code context is sent to the LLM API for test generation.

## Core Architecture

```
apptest CLI (runs in customer's CI)
  │
  ├── analyze   → reads PR diff, extracts relevant source context
  ├── generate  → sends context to LLM, receives test cases
  ├── execute   → runs tests on device via ADB
  └── report    → posts results to PR comment, outputs JSON
```

### What Runs Where

**Customer's CI environment:**
- apptest CLI (thin client)
- Source code (never leaves their infra)
- Git diff computation (local)
- Test execution on their devices/emulators
- Screenshots and logs

**LLM API (Anthropic):**
- Receives: relevant code snippets, PR context (NOT the full repo)
- Returns: generated test cases, action decisions during execution

## Technology Stack

- **Language:** Python 3.11+
- **CLI framework:** Click
- **Device interaction:** ADB (via subprocess or pure-adb-python)
- **UI inspection:** UIAutomator (hierarchy dump via ADB)
- **OCR:** PaddleOCR (for elements missing from hierarchy)
- **LLM:** Anthropic Claude API (claude-sonnet-4-20250514)
- **Distribution:** pip package (`pip install apptest`)
- **Config format:** YAML

## CLI Commands

### `apptest init`

One-time codebase scan that generates an app profile. Run this when first setting up AppTest for a project, or after major architecture changes.

```bash
apptest init --repo .
```

**What it does:**

1. Detect project structure (single-module vs multi-module Gradle)
2. Find all source roots and resource directories
3. Discover all screen files (Activities, Fragments, Composables)
4. Find associated layouts, ViewModels, and Repositories for each screen
5. Detect architecture pattern (MVVM, MVI, MVP, Compose, hybrid)
6. Detect navigation type (Jetpack Navigation, Compose Navigation, manual)
7. Detect DI framework (Hilt, Dagger, Koin, manual)
8. Trace full dependency chains for every screen
9. Generate `.apptest/app-profile.yml`
10. Print summary for human review

**CLI output:**

```
$ apptest init --repo .

Architecture: MVVM
```

**App Profile Scanner (`scanner/project_scanner.py`):**

```python
"""
One-time codebase scanner that generates the app profile.
Discovers project structure, architecture, screens, and dependency chains.
"""

def scan_project(repo_path: str) -> dict:
    """Full project scan to generate app-profile.yml."""
    profile = {"auto": {}, "overrides": {}}
    auto = profile["auto"]
    
    # 1. Detect project structure
    auto["project_structure"] = detect_project_structure(repo_path)
    source_roots = auto["project_structure"]["source_roots"]
    
    # 2. Detect architecture
    auto["architecture"] = detect_architecture(source_roots, repo_path)
    
    # 3. Find all screens
    auto["screens"] = discover_all_screens(source_roots, repo_path)
    
    # 4. Trace dependency chains for each screen
    auto["dependency_chains"] = trace_all_chains(auto["screens"], repo_path)
    
    # 5. Detect navigation
    auto["navigation"] = detect_navigation_type(repo_path)
    
    # 6. Detect DI
    auto["dependency_injection"] = detect_di_framework(repo_path)
    
    # 7. Timestamp
    auto["last_updated"] = datetime.now().isoformat()
    
    # Initialize empty overrides section for human edits
    profile["overrides"] = {
        "additional_screens": [],
        "reclassifications": [],
        "chain_overrides": [],
        "ignore": []
    }
    
    return profile


def detect_project_structure(repo_path: str) -> dict:
    """Detect single-module vs multi-module and find source roots."""
    settings_file = None
    for name in ["settings.gradle.kts", "settings.gradle"]:
        path = os.path.join(repo_path, name)
        if os.path.exists(path):
            settings_file = path
            break
    
    modules = []
    if settings_file:
        modules = parse_gradle_modules(settings_file)
    
    if not modules:
        modules = ["."]
    
    source_roots = []
    for module in modules:
        for src_variant in ["src/main/java", "src/main/kotlin"]:
            src_path = os.path.join(repo_path, module, src_variant)
            if os.path.isdir(src_path):
                source_roots.append(os.path.relpath(src_path, repo_path))
    
    resource_roots = []
    for module in modules:
        res_path = os.path.join(repo_path, module, "src/main/res")
        if os.path.isdir(res_path):
            resource_roots.append(os.path.relpath(res_path, repo_path))
    
    return {
        "type": "multi-module" if len(modules) > 1 else "single-module",
        "modules": modules,
        "source_roots": source_roots,
        "resource_roots": resource_roots
    }


def detect_architecture(source_roots: list, repo_path: str) -> dict:
    """Detect app architecture pattern from code patterns."""
    has_viewmodels = False
    has_compose = False
    has_presenters = False
    has_mvi_pattern = False
    screen_types_found = set()
    
    for root in source_roots:
        for kt_file in find_kotlin_files(repo_path, root):
            content = read_file_safe(repo_path, kt_file)
            if not content:
                continue
            
            filename = os.path.basename(kt_file)
            
            if "ViewModel" in filename:
                has_viewmodels = True
            if "@Composable" in content:
                has_compose = True
            if "Presenter" in filename:
                has_presenters = True
            if "sealed class" in content and any(kw in content for kw in ["Intent", "UiState", "Effect", "Action"]):
                has_mvi_pattern = True
            
            if "Fragment" in filename and ": Fragment(" in content:
                screen_types_found.add("Fragment")
            if "Activity" in filename and "Activity()" in content:
                screen_types_found.add("Activity")
    
    if has_compose and "Fragment" not in screen_types_found:
        pattern = "compose"
    elif has_mvi_pattern and has_viewmodels:
        pattern = "mvi"
    elif has_viewmodels:
        pattern = "mvvm"
    elif has_presenters:
        pattern = "mvp"
    else:
        pattern = "unknown"
    
    return {
        "pattern": pattern,
        "has_compose": has_compose,
        "screen_types_found": list(screen_types_found)
    }


def discover_all_screens(source_roots: list, repo_path: str) -> list:
    """Find all screen files and their associations."""
    screens = []
    
    for root in source_roots:
        for kt_file in find_kotlin_files(repo_path, root):
            content = read_file_safe(repo_path, kt_file)
            if not content:
                continue
            
            if not is_screen_file(content):
                continue
            
            screen = {
                "name": extract_class_name(kt_file, repo_path),
                "file": kt_file,
                "type": detect_screen_type(content)
            }
            
            # Find associated layout
            layout = find_layout_for_screen(kt_file, repo_path)
            if layout:
                screen["layout"] = layout
            
            # Find associated ViewModel
            vm = find_viewmodel_in_source(content, kt_file, repo_path)
            if vm:
                screen["viewmodel"] = vm
            
            screens.append(screen)
    
    return screens


def is_screen_file(content: str) -> bool:
    """Determine if a file defines a UI screen."""
    screen_indicators = [
        ": Fragment(",
        ": AppCompatActivity(",
        ": ComponentActivity(",
        ": Activity(",
        ": DialogFragment(",
        ": BottomSheetDialogFragment(",
        "@Composable\nfun",    # top-level composable
        "setContent {",         # Compose activity
        "NavHost(",
    ]
    return any(indicator in content for indicator in screen_indicators)


def detect_screen_type(content: str) -> str:
    """Detect what kind of screen this is."""
    if ": Fragment(" in content or ": DialogFragment(" in content:
        return "Fragment"
    if ": BottomSheetDialogFragment(" in content:
        return "BottomSheet"
    if "Activity(" in content:
        return "Activity"
    if "@Composable" in content:
        return "Composable"
    return "unknown"


def find_viewmodel_in_source(content: str, screen_file: str, repo_path: str) -> str | None:
    """Find the ViewModel class referenced in a screen file."""
    # Common patterns:
    # val viewModel: SearchViewModel by viewModels()
    # private val viewModel by viewModels<SearchViewModel>()
    # @Inject lateinit var viewModel: SearchViewModel
    
    patterns = [
        r'by\s+viewModels\s*<\s*(\w+ViewModel)\s*>',
        r'by\s+viewModels\s*\(\s*\)\s*.*?:\s*(\w+ViewModel)',
        r':\s+(\w+ViewModel)\s+by',
        r':\s+(\w+ViewModel)\s*$',
        r'viewModels\s*<\s*(\w+ViewModel)\s*>',
        r'(\w+ViewModel)\s+by\s+viewModels',
        r'(\w+Presenter)\s+',
    ]
    
    for pattern in patterns:
        match = re.search(pattern, content, re.MULTILINE)
        if match:
            return match.group(1)
    
    return None


def trace_all_chains(screens: list, repo_path: str) -> list:
    """Build full dependency chains for every screen."""
    chains = []
    
    for screen in screens:
        chain_members = [screen["name"]]
        
        # Add ViewModel
        vm_name = screen.get("viewmodel")
        if vm_name:
            chain_members.append(vm_name)
            
            # Find what the ViewModel depends on (Repositories, UseCases)
            vm_file = find_file_for_class(vm_name, repo_path)
            if vm_file:
                vm_content = read_file_safe(repo_path, vm_file)
                if vm_content:
                    deps = extract_constructor_dependencies(vm_content)
                    chain_members.extend(deps)
                    
                    # Go one more level: what do Repositories depend on?
                    for dep in deps:
                        dep_file = find_file_for_class(dep, repo_path)
                        if dep_file:
                            dep_content = read_file_safe(repo_path, dep_file)
                            if dep_content:
                                sub_deps = extract_constructor_dependencies(dep_content)
                                chain_members.extend(sub_deps)
        
        chains.append({
            "screen": screen["name"],
            "chain": chain_members
        })
    
    return chains


def extract_constructor_dependencies(content: str) -> list:
    """
    Extract class dependencies from constructor injection.
    Handles both Hilt/Dagger @Inject constructor and Koin patterns.
    """
    deps = []
    
    # @Inject constructor(private val repo: SearchRepository, ...)
    constructor_match = re.search(
        r'(?:@Inject\s+)?constructor\s*\((.*?)\)',
        content, re.DOTALL
    )
    
    if constructor_match:
        params = constructor_match.group(1)
        # Extract type names from parameters
        type_matches = re.findall(r':\s*(\w+(?:Repository|DataSource|UseCase|Interactor|Service|Api|Manager|Provider))', params)
        deps.extend(type_matches)
    
    # Also check for property injection
    prop_matches = re.findall(
        r'(?:@Inject|private\s+val|val)\s+\w+\s*:\s*(\w+(?:Repository|DataSource|UseCase|Interactor|Service|Api|Manager|Provider))',
        content
    )
    deps.extend(prop_matches)
    
    return list(set(deps))
```

**Profile Manager (`scanner/profile_manager.py`):**

The profile manager handles loading the profile with overrides applied, and merging auto-detected data with human-curated overrides.

```python
"""
Load, merge, and save the app profile.
Handles the two-layer system: auto-detected data + human overrides.
"""

def load_effective_profile(repo_path: str) -> dict:
    """
    Load profile with human overrides applied on top of auto-detected data.
    This is what the analyzer uses at runtime.
    """
    profile_path = os.path.join(repo_path, ".apptest/app-profile.yml")
    
    if not os.path.exists(profile_path):
        return None  # need to run apptest init first
    
    raw = load_yaml(profile_path)
    auto = raw.get("auto", {})
    overrides = raw.get("overrides", {})
    
    # Start with auto-detected screens
    screens = list(auto.get("screens", []))
    
    # Add manually specified screens
    for additional in overrides.get("additional_screens", []):
        # Don't add if already exists by name
        if not any(s["name"] == additional["name"] for s in screens):
            additional["source"] = "manual"
            screens.append(additional)
    
    # Build reclassification lookup
    reclassifications = {}
    for reclass in overrides.get("reclassifications", []):
        reclassifications[reclass["file"]] = reclass["actual_type"]
    
    # Build dependency chains with overrides applied
    chains = list(auto.get("dependency_chains", []))
    for override in overrides.get("chain_overrides", []):
        # Replace the auto chain with the manual one
        chains = [c for c in chains if c["screen"] != override["screen"]]
        override["source"] = "manual"
        chains.append(override)
    
    # Remove ignored files
    ignored = set(overrides.get("ignore", []))
    screens = [s for s in screens if s.get("file") not in ignored]
    
    return {
        "screens": screens,
        "dependency_chains": chains,
        "architecture": auto.get("architecture", {}),
        "navigation": auto.get("navigation", {}),
        "project_structure": auto.get("project_structure", {}),
        "dependency_injection": auto.get("dependency_injection", {}),
        "reclassifications": reclassifications,
        "ignored_files": ignored
    }


def lookup_affected_screens(file_path: str, profile: dict) -> list:
    """
    Look up which screens a file affects using the pre-built dependency chains.
    Much faster and more reliable than tracing at analysis time.
    """
    class_name = extract_class_name_from_path(file_path)
    
    affected = []
    for chain in profile["dependency_chains"]:
        if class_name in chain["chain"]:
            affected.append({
                "screen": chain["screen"],
                "chain": chain["chain"],
                "confidence": "high"
            })
    
    # If not found in any chain, this file might be new
    if not affected:
        return [{
            "screen": "unknown",
            "chain": [class_name],
            "confidence": "low",
            "note": "File not found in any known dependency chain. May need profile update."
        }]
    
    return affected
```

#### Profile Auto-Update (`analyzer/profile_updater.py`)

Every `apptest analyze` run auto-updates the profile's auto section based on what changed in the PR. Human overrides are never touched.

```python
"""
Auto-update the app profile based on changes found in a PR.
Updates the 'auto' section only — never modifies 'overrides'.
"""

def update_profile_from_pr(profile_path: str, changed_files: list, repo_path: str):
    """
    Patch the auto section of the profile based on this PR's changes.
    Called at the end of every apptest analyze run.
    """
    raw = load_yaml(profile_path)
    auto = raw.get("auto", {})
    screens = auto.get("screens", [])
    chains = auto.get("dependency_chains", [])
    
    for file_path in changed_files:
        content = read_file_safe(repo_path, file_path)
        
        if content is None:
            # File was deleted — remove from auto section
            _remove_file_from_auto(screens, chains, file_path)
            continue
        
        if not file_path.endswith((".kt", ".java")):
            continue
        
        class_name = extract_class_name(file_path, repo_path)
        
        # Check if this is a new or modified screen
        if is_screen_file(content):
            existing = next((s for s in screens if s["name"] == class_name), None)
            
            if existing:
                # Update existing screen entry
                existing["file"] = file_path
                new_layout = find_layout_for_screen(file_path, repo_path)
                if new_layout:
                    existing["layout"] = new_layout
                new_vm = find_viewmodel_in_source(content, file_path, repo_path)
                if new_vm:
                    existing["viewmodel"] = new_vm
            else:
                # New screen discovered — add it
                new_screen = {
                    "name": class_name,
                    "file": file_path,
                    "type": detect_screen_type(content),
                    "auto_discovered_in_pr": True
                }
                layout = find_layout_for_screen(file_path, repo_path)
                if layout:
                    new_screen["layout"] = layout
                vm = find_viewmodel_in_source(content, file_path, repo_path)
                if vm:
                    new_screen["viewmodel"] = vm
                screens.append(new_screen)
        
        # Check if this file should be part of a dependency chain
        if is_logic_file(file_path):
            _update_chains_for_file(chains, class_name, file_path, repo_path)
    
    # Update timestamp
    auto["last_updated"] = datetime.now().isoformat()
    auto["screens"] = screens
    auto["dependency_chains"] = chains
    
    # Save — preserving the overrides section untouched
    raw["auto"] = auto
    save_yaml(profile_path, raw)


def _remove_file_from_auto(screens: list, chains: list, deleted_file: str):
    """Remove references to a deleted file."""
    class_name = extract_class_name_from_path(deleted_file)
    
    # Remove from screens
    screens[:] = [s for s in screens if s.get("file") != deleted_file]
    
    # Remove from chains
    for chain in chains:
        if class_name in chain["chain"]:
            chain["chain"].remove(class_name)
    
    # Remove empty chains
    chains[:] = [c for c in chains if len(c.get("chain", [])) > 1]


def _update_chains_for_file(chains: list, class_name: str, file_path: str, repo_path: str):
    """Update dependency chains if a new logic file is relevant."""
    # Check if this class is already in a chain
    for chain in chains:
        if class_name in chain["chain"]:
            return  # already tracked
    
    # Not in any chain — try to trace it
    content = read_file_safe(repo_path, file_path)
    if not content:
        return
    
    deps = extract_constructor_dependencies(content)
    # Find if any dependency is already in a chain
    for chain in chains:
        for dep in deps:
            if dep in chain["chain"]:
                # This file depends on something in an existing chain
                # Insert it in the right position
                dep_idx = chain["chain"].index(dep)
                chain["chain"].insert(dep_idx, class_name)
                return
    
    # Also check if any existing chain member depends on this new class
    consumers = find_consumers(class_name, repo_path, 
                              target_types=["ViewModel", "Presenter", "Fragment", "Activity"])
    for chain in chains:
        for consumer_file in consumers:
            consumer_name = extract_class_name_from_path(consumer_file)
            if consumer_name in chain["chain"]:
                consumer_idx = chain["chain"].index(consumer_name)
                chain["chain"].insert(consumer_idx + 1, class_name)
                return
```

### `apptest analyze`

Reads a PR diff, classifies each change, traces logic changes to their UI screens, and extracts relevant context for test generation.

```bash
apptest analyze --diff "HEAD~1..HEAD" --repo . --config apptest.yml
```

**What it does:**

1. Load the app profile (`.apptest/app-profile.yml`) generated by `apptest init`
2. Run `git diff` to get changed files
3. Parse the diff to identify which files changed and how
4. Classify each changed file into one of four categories:
   - **UI changes** — layout XML, navigation graph, drawable, style, string resources
   - **Logic changes** — ViewModel, Repository, UseCase, Service, Adapter, Model, any Kotlin/Java business code
   - **Test changes** — existing test files (informative signal, don't generate tests for tests)
   - **Infrastructure changes** — build config, CI, dependencies, manifest-only (usually don't need functional tests)
5. For UI changes: extract layout content, string resources, navigation connections
6. For logic changes: use the profile's pre-built dependency chains to quickly look up which screen is affected, read the code diff AND full file for context
7. Classify the nature of each logic change: new feature, bug fix, refactor, error handling improvement, performance change
8. Parse AndroidManifest.xml for the full Activity inventory
9. Auto-update the profile's auto section with any new screens, chains, or deletions found in this PR
10. Output analysis to the run directory

#### Change Classification (`change_classifier.py`)

Every changed file gets classified so the generator can produce the right kind of tests.

```python
"""
Classify changed files into categories that determine
how the generator should handle them.
"""

def classify_file(file_path: str) -> str:
    """Classify a changed file by its role in the app."""
    
    # UI resource files
    if file_path.startswith("res/layout") or "/res/layout" in file_path:
        return "ui_layout"
    if file_path.startswith("res/navigation") or "/res/navigation" in file_path:
        return "ui_navigation"
    if "res/values/strings" in file_path:
        return "ui_strings"
    if "res/drawable" in file_path or "res/mipmap" in file_path:
        return "ui_resource"
    if "res/menu" in file_path:
        return "ui_menu"
    if "res/values/styles" in file_path or "res/values/themes" in file_path:
        return "ui_style"
    
    # Test files
    if "/test/" in file_path or "/androidTest/" in file_path:
        return "test"
    
    # Infrastructure files
    if file_path.endswith((".gradle", ".gradle.kts", ".properties")):
        return "infra_build"
    if file_path.endswith((".yml", ".yaml")) and ("ci" in file_path.lower() or "workflow" in file_path.lower()):
        return "infra_ci"
    if file_path == "AndroidManifest.xml" or file_path.endswith("AndroidManifest.xml"):
        return "infra_manifest"
    
    # Logic/code files — the most common and most important case
    if file_path.endswith((".kt", ".java")):
        return classify_code_file(file_path)
    
    return "other"


def classify_code_file(file_path: str) -> str:
    """
    Classify a Kotlin/Java file by its architectural role.
    This determines how we trace it to the UI and what kind
    of tests are appropriate.
    """
    filename = file_path.split("/")[-1]
    
    # Direct UI code
    if "Activity" in filename:
        return "logic_screen"
    if "Fragment" in filename:
        return "logic_screen"
    if "Composable" in filename or "Screen" in filename:
        return "logic_screen"
    
    # ViewModel layer — one hop from the screen
    if "ViewModel" in filename:
        return "logic_viewmodel"
    if "Presenter" in filename:
        return "logic_viewmodel"
    
    # Data layer — two hops from the screen
    if "Repository" in filename:
        return "logic_repository"
    if "DataSource" in filename:
        return "logic_datasource"
    if "UseCase" in filename or "Interactor" in filename:
        return "logic_usecase"
    
    # Service/infrastructure layer
    if "Service" in filename and "Api" not in filename:
        return "logic_service"
    if "Api" in filename or "ApiService" in filename:
        return "logic_api"
    if "Retrofit" in filename or "OkHttp" in filename:
        return "logic_network"
    
    # Data models
    if "Model" in filename or "Entity" in filename or "Dto" in filename:
        return "logic_model"
    if "Response" in filename or "Request" in filename:
        return "logic_model"
    
    # Utility code
    if "Util" in filename or "Helper" in filename or "Extension" in filename:
        return "logic_util"
    if "Mapper" in filename or "Converter" in filename:
        return "logic_util"
    
    # Adapters (RecyclerView, etc.)
    if "Adapter" in filename or "ViewHolder" in filename:
        return "logic_adapter"
    
    # Default: treat as logic code
    return "logic_other"


def classify_change_nature(diff_content: str) -> str:
    """
    Classify what KIND of change was made (bug fix, new feature, etc).
    This helps the generator produce the right type of tests.
    """
    lines = diff_content.split("\n")
    added = [l[1:] for l in lines if l.startswith("+") and not l.startswith("+++")]
    removed = [l[1:] for l in lines if l.startswith("-") and not l.startswith("---")]
    diff_lower = diff_content.lower()
    
    # Pure addition — likely new feature
    if len(added) > 0 and len(removed) == 0:
        return "new_feature"
    
    # Pure deletion — feature removal
    if len(removed) > 0 and len(added) == 0:
        return "feature_removal"
    
    # Error handling added
    if any(kw in diff_lower for kw in ["try {", "catch", "exception", "error", "throw", "result.failure"]):
        if sum(1 for a in added if any(kw in a.lower() for kw in ["try", "catch", "exception"])) > \
           sum(1 for r in removed if any(kw in r.lower() for kw in ["try", "catch", "exception"])):
            return "error_handling"
    
    # Bug fix signals
    if any(kw in diff_lower for kw in ["fix", "bug", "crash", "null", "npe", "workaround"]):
        return "bug_fix"
    
    # Validation changes
    if any(kw in diff_lower for kw in ["valid", "check", "require", "assert", "verify"]):
        return "validation"
    
    # Performance changes
    if any(kw in diff_lower for kw in ["cache", "lazy", "async", "coroutine", "optimize", "batch"]):
        return "performance"
    
    # Refactor — similar amount added and removed, structural change
    if 0.5 < len(added) / max(len(removed), 1) < 2.0:
        return "refactor"
    
    return "modification"
```

#### Dependency Tracing (`dependency_tracer.py`)

For logic changes (ViewModel, Repository, UseCase, etc.), we need to trace through the dependency chain to find which screen is affected. This is the critical step that connects a backend code change to a user-facing screen.

```python
"""
Trace code dependencies to find which UI screens are affected
by a change in business logic code.

Uses text-based search (grep-style), not AST analysis.
This works because Android projects follow naming conventions:
- SearchViewModel is referenced by name in SearchFragment
- ArticleRepository is injected into ArticleViewModel
- UserService is called from UserRepository

Not perfect, but catches 90%+ of real dependency chains.
"""

import os
import re


def trace_to_screen(file_path: str, file_type: str, repo_path: str) -> dict:
    """
    Trace a changed file to the screen(s) it affects.
    Returns the dependency chain and the final screen files.
    
    Examples:
        Repository -> ViewModel -> Fragment
        UseCase -> ViewModel -> Fragment  
        ApiService -> Repository -> ViewModel -> Fragment
        Model -> (many files) -> pick the ones closest to UI
        Adapter -> Fragment (direct, adapters are used in a specific screen)
    """
    class_name = extract_class_name(file_path, repo_path)
    
    if file_type == "logic_screen":
        # Already a screen file — no tracing needed
        return {
            "chain": [file_path],
            "screen_files": [file_path],
            "confidence": "high"
        }
    
    if file_type == "logic_viewmodel":
        # One hop: find which Fragment/Activity uses this ViewModel
        screens = find_consumers(class_name, repo_path, target_types=["Fragment", "Activity", "Composable"])
        return {
            "chain": [file_path] + screens,
            "screen_files": screens,
            "confidence": "high" if screens else "low"
        }
    
    if file_type in ("logic_repository", "logic_datasource", "logic_usecase"):
        # Two hops: find ViewModel consumers, then find their screens
        viewmodels = find_consumers(class_name, repo_path, target_types=["ViewModel", "Presenter"])
        screens = []
        for vm_file in viewmodels:
            vm_class = extract_class_name(vm_file, repo_path)
            vm_screens = find_consumers(vm_class, repo_path, target_types=["Fragment", "Activity", "Composable"])
            screens.extend(vm_screens)
        return {
            "chain": [file_path] + viewmodels + screens,
            "screen_files": list(set(screens)),
            "confidence": "high" if screens else "medium"
        }
    
    if file_type == "logic_api":
        # Three hops: API -> Repository -> ViewModel -> Screen
        repos = find_consumers(class_name, repo_path, target_types=["Repository", "DataSource"])
        viewmodels = []
        for repo_file in repos:
            repo_class = extract_class_name(repo_file, repo_path)
            vms = find_consumers(repo_class, repo_path, target_types=["ViewModel", "Presenter"])
            viewmodels.extend(vms)
        screens = []
        for vm_file in set(viewmodels):
            vm_class = extract_class_name(vm_file, repo_path)
            vm_screens = find_consumers(vm_class, repo_path, target_types=["Fragment", "Activity", "Composable"])
            screens.extend(vm_screens)
        return {
            "chain": [file_path] + repos + list(set(viewmodels)) + list(set(screens)),
            "screen_files": list(set(screens)),
            "confidence": "medium" if screens else "low"
        }
    
    if file_type == "logic_model":
        # Models can be used anywhere — find the closest UI consumers
        all_consumers = find_consumers(class_name, repo_path)
        # Prefer ViewModel/Fragment consumers over Repository/Util consumers
        screen_consumers = [f for f in all_consumers if any(t in f for t in ["Fragment", "Activity", "ViewModel"])]
        return {
            "chain": [file_path] + screen_consumers,
            "screen_files": [f for f in screen_consumers if "Fragment" in f or "Activity" in f],
            "confidence": "medium"
        }
    
    if file_type == "logic_adapter":
        # Adapters are usually in the same package as their Fragment
        screens = find_consumers(class_name, repo_path, target_types=["Fragment", "Activity"])
        return {
            "chain": [file_path] + screens,
            "screen_files": screens,
            "confidence": "high" if screens else "medium"
        }
    
    # Default: broad search
    consumers = find_consumers(class_name, repo_path)
    screens = [f for f in consumers if "Fragment" in f or "Activity" in f]
    return {
        "chain": [file_path] + screens,
        "screen_files": screens,
        "confidence": "low"
    }


def extract_class_name(file_path: str, repo_path: str) -> str:
    """
    Extract the primary class/interface name from a file.
    Reads the file and finds the class declaration.
    Falls back to deriving from filename.
    """
    full_path = os.path.join(repo_path, file_path)
    
    try:
        with open(full_path) as f:
            content = f.read()
        
        # Kotlin: class/object/interface declaration
        match = re.search(r'(?:class|object|interface)\s+(\w+)', content)
        if match:
            return match.group(1)
    except FileNotFoundError:
        pass
    
    # Fallback: derive from filename
    filename = os.path.basename(file_path)
    return filename.replace(".kt", "").replace(".java", "")


def find_consumers(class_name: str, repo_path: str, 
                   target_types: list[str] = None) -> list[str]:
    """
    Find files that reference a given class name.
    Optionally filter to only files matching target_types in their name.
    
    Uses grep-style text search. Simple but effective for convention-following
    Android projects.
    """
    consumers = []
    
    for root, dirs, files in os.walk(repo_path):
        # Skip build, test, and hidden directories
        dirs[:] = [d for d in dirs if d not in [
            "build", ".gradle", ".git", "node_modules", 
            "test", "androidTest", ".idea"
        ]]
        
        for filename in files:
            if not filename.endswith((".kt", ".java")):
                continue
            
            file_path = os.path.join(root, filename)
            rel_path = os.path.relpath(file_path, repo_path)
            
            # Skip the file itself
            if class_name in filename and filename.replace(".kt", "").replace(".java", "") == class_name:
                continue
            
            # Filter by target type if specified
            if target_types:
                if not any(t in filename for t in target_types):
                    continue
            
            try:
                with open(file_path) as f:
                    content = f.read()
                if class_name in content:
                    consumers.append(rel_path)
            except (FileNotFoundError, UnicodeDecodeError):
                continue
    
    return consumers
```

#### Context Builder (`context_builder.py`)

Builds the right context based on what type of change each file represents. UI changes get layout and string context. Logic changes get code context and dependency chains.

```python
"""
Build analysis context for each changed file based on its type.
Different change types need different context for effective test generation.
"""

def build_analysis_context(changed_files: list, diff_content: str, repo_path: str) -> dict:
    """
    Build the complete analysis context from a PR diff.
    Classifies each change and gathers appropriate context.
    """
    context = {
        "ui_changes": [],
        "logic_changes": [],
        "test_changes": [],
        "infra_changes": []
    }
    
    for file_path in changed_files:
        file_diff = extract_file_diff(diff_content, file_path)
        file_type = classify_file(file_path)
        
        # UI resource changes
        if file_type.startswith("ui_"):
            context["ui_changes"].append(build_ui_context(
                file_path, file_diff, file_type, repo_path
            ))
        
        # Logic/code changes — the most common case
        elif file_type.startswith("logic_"):
            context["logic_changes"].append(build_logic_context(
                file_path, file_diff, file_type, repo_path
            ))
        
        # Existing test changes
        elif file_type == "test":
            context["test_changes"].append({
                "file": file_path,
                "diff": file_diff,
                "note": "Existing test was modified — indicates what behavior changed"
            })
        
        # Infrastructure changes
        else:
            context["infra_changes"].append({
                "file": file_path,
                "diff": file_diff,
                "type": file_type
            })
    
    return context


def build_ui_context(file_path: str, diff: str, file_type: str, repo_path: str) -> dict:
    """Build context for a UI resource change."""
    ctx = {
        "file": file_path,
        "diff": diff,
        "type": file_type,
        "content": read_file_safe(repo_path, file_path)
    }
    
    if file_type == "ui_layout":
        # Find which screen uses this layout
        screen = map_layout_to_screen(file_path, repo_path)
        ctx["affected_screen"] = screen
        ctx["related_strings"] = find_string_refs_in_layout(file_path, repo_path)
    
    elif file_type == "ui_strings":
        # Parse old vs new strings to find additions and changes
        old_strings, new_strings = diff_strings(diff)
        ctx["new_strings"] = new_strings
        ctx["changed_strings"] = find_changed_strings(old_strings, new_strings)
        ctx["removed_strings"] = find_removed_strings(old_strings, new_strings)
    
    elif file_type == "ui_navigation":
        ctx["nav_graph_content"] = read_file_safe(repo_path, file_path)
    
    return ctx


def build_logic_context(file_path: str, diff: str, file_type: str, repo_path: str) -> dict:
    """
    Build context for a logic/code change.
    This includes the code itself, the dependency chain to the UI,
    and classification of the change nature.
    """
    # Read the full file for context (not just the diff)
    full_source = read_file_safe(repo_path, file_path)
    
    # Trace to the screen this code affects
    trace = trace_to_screen(file_path, file_type, repo_path)
    
    # Classify what kind of change this is
    change_nature = classify_change_nature(diff)
    
    # For each affected screen, gather its layout and strings too
    screen_context = []
    for screen_file in trace["screen_files"]:
        layout = find_layout_for_screen(screen_file, repo_path)
        screen_context.append({
            "screen_file": screen_file,
            "screen_source": read_file_safe(repo_path, screen_file),
            "layout": read_file_safe(repo_path, layout) if layout else None,
            "layout_file": layout
        })
    
    return {
        "file": file_path,
        "diff": diff,
        "full_source": full_source,
        "type": file_type,
        "change_nature": change_nature,
        "dependency_chain": trace["chain"],
        "affected_screens": trace["screen_files"],
        "trace_confidence": trace["confidence"],
        "screen_context": screen_context
    }
```

### `apptest generate`

Takes the analysis output and generates natural language test cases using the LLM. Handles both UI changes (where layouts and strings changed) and logic changes (where only code changed) with different prompting strategies.

```bash
apptest generate --analysis .apptest/analysis.json --output .apptest/tests.json
```

**What it does:**

1. Load the analysis JSON
2. Load any existing knowledge base for this app
3. Build a prompt that includes BOTH UI and logic change context:
   - For UI changes: layout diffs, new/changed strings, navigation connections
   - For logic changes: the code diff, full source file, dependency chain to screen, change nature classification
   - For test changes: what existing tests were modified (signals what behavior changed)
   - Knowledge base entries for affected screens
4. Call the LLM to generate test cases
5. Parse and validate the response
6. Output test cases to `.apptest/tests.json`

**Prompt template (in `prompts.py`):**

The prompt adapts based on what types of changes are present. Logic changes include the actual code diff so the LLM can reason about behavioral impact.

```python
TEST_GENERATION_PROMPT = """
You are a senior QA engineer generating test cases for a mobile app update.
Read all changes carefully and generate tests from the USER's perspective —
every test must be described as actions a human tester would perform on the device.

## Changes in This PR

{formatted_changes}

## What We Know About This App From Previous Testing

{knowledge_base_summary}

## Instructions

Generate test cases that cover:
1. Happy path for any new or modified features
2. Error cases — look for error strings, catch blocks, Result.failure, exception handling
3. Edge cases (empty input, boundary values, null states, network failure, rapid interactions)
4. Regression — verify existing functionality on modified screens still works

For bug fixes: write a test that would have CAUGHT the original bug
For new features: cover happy path + error cases + edge cases
For refactors: write regression tests to verify behavior didn't change
For error handling changes: test both the error path AND the happy path
For performance changes: test the affected flow works correctly (timing is hard to test via UI)

CRITICAL RULES:
- Tests must be user-facing actions, NOT code-level assertions
- BAD: "Verify the API returns 200" or "Assert repository returns Result.success"
- GOOD: "Search for 'Einstein', verify results appear within a few seconds"
- Each test description must be concrete step-by-step instructions
- Include what the expected outcome should look like to the user

Respond as JSON array:
[
  {{
    "id": "test_001",
    "description": "step by step natural language test description",
    "covers": "what aspect of the change this tests",
    "change_type": "new_feature|bug_fix|regression|error_case|edge_case",
    "priority": "high|medium|low",
    "test_data": {{"key": "value"}}
  }}
]
"""


def format_changes(context: dict) -> str:
    """
    Format all changes into a single string for the LLM prompt.
    Adapts format based on change type.
    """
    sections = []
    
    # UI changes — show layout diffs and string changes
    if context["ui_changes"]:
        sections.append("### UI Changes\n")
        for change in context["ui_changes"]:
            sections.append(f"**File:** {change['file']} (type: {change['type']})")
            if change.get("affected_screen"):
                sections.append(f"**Affects screen:** {change['affected_screen']}")
            if change.get("related_strings"):
                sections.append(f"**Related strings:** {json.dumps(change['related_strings'], indent=2)}")
            if change.get("new_strings"):
```

### `apptest execute`

Runs generated tests on a device or emulator.

```bash
apptest execute --tests .apptest/tests.json --device emulator-5554 --app org.wikipedia
```

**What it does for each test:**

1. Reset app state (`adb shell pm clear <package>`)
2. Launch the app (`adb shell am start -n <package>/<main_activity>`)
3. Wait for app to be ready
4. Call the LLM planner to decompose the test description into goals
5. For each goal, run the execution loop:
   a. Dump UI hierarchy via UIAutomator
   b. Run OCR on screenshot to find elements missing from hierarchy
   c. Merge both element sources
   d. Check if goal is already achieved (success criteria met)
   e. Check knowledge base for known action
   f. If unknown, call LLM with current screen elements + goal to decide next action
   g. Execute the action via ADB
   h. Record what happened for the knowledge base
6. Capture screenshot on pass or fail
7. Record result

**The execution loop in detail:**

```python
def execute_goal(device, goal, test_data, knowledge_base, max_steps=15):
    """
    Execute a single goal on the device.
    
    Args:
        device: ADB device connection
        goal: dict with 'description' and 'success_criteria'
        test_data: dict of test parameters
        knowledge_base: AppKnowledge instance
        max_steps: safety limit to prevent infinite loops
    
    Returns:
        GoalResult with success, actions taken, screenshots
    """
    actions_taken = []
    
    for step in range(max_steps):
        # 1. Observe current state
        hierarchy = dump_ui_hierarchy(device)
        screenshot_path = capture_screenshot(device)
        ocr_results = run_ocr(screenshot_path)
        elements = merge_hierarchy_and_ocr(hierarchy, ocr_results)
        
        # 2. Check if goal is achieved
        if check_success_criteria(goal['success_criteria'], elements, hierarchy):
            knowledge_base.record_success(goal, actions_taken)
            return GoalResult(success=True, actions=actions_taken)
        
        # 3. Check knowledge base for known action
        known_action = knowledge_base.lookup(
            goal=goal['description'],
            current_screen=identify_screen(hierarchy),
            elements=elements
        )
        
        if known_action:
            execute_action(device, known_action)
            actions_taken.append(known_action)
            time.sleep(0.5)
            continue
        
        # 4. Ask LLM what to do
        action = llm_decide_action(
            goal=goal,
            elements=elements,
            hierarchy_activity=hierarchy.activity_name,
            test_data=test_data
        )
        
        execute_action(device, action)
        actions_taken.append(action)
        
        # 5. Record for knowledge base
        knowledge_base.record_action(
            goal=goal['description'],
            screen=identify_screen(hierarchy),
            action=action
        )
        
        time.sleep(0.5)
    
    return GoalResult(success=False, reason="max_steps_exceeded", actions=actions_taken)
```

```python
"""
UIAutomator hierarchy dump and parsing.
Extracts interactive elements with their properties.
"""

import xml.etree.ElementTree as ET

class UIElement:
    """Represents a single UI element from the hierarchy."""
    def __init__(self, node):
        self.text = node.get("text", "")
        self.resource_id = node.get("resource-id", "")
        self.class_name = node.get("class", "")
        self.content_desc = node.get("content-desc", "")
        self.clickable = node.get("clickable") == "true"
        self.checkable = node.get("checkable") == "true"
        self.checked = node.get("checked") == "true"
        self.enabled = node.get("enabled") == "true"
        self.scrollable = node.get("scrollable") == "true"
        self.focusable = node.get("focusable") == "true"
        self.selected = node.get("selected") == "true"
        self.bounds = self._parse_bounds(node.get("bounds", ""))
        self.package = node.get("package", "")
    
    def _parse_bounds(self, bounds_str):
        """Parse '[0,0][1080,2400]' format into dict."""
        match = re.findall(r"\[(\d+),(\d+)\]", bounds_str)
        if len(match) == 2:
            return {
                "left": int(match[0][0]),
                "top": int(match[0][1]),
                "right": int(match[1][0]),
                "bottom": int(match[1][1])
            }
        return None
    
    def center(self) -> tuple[int, int]:
        """Get center coordinates of this element."""
        b = self.bounds
        return (b["left"] + b["right"]) // 2, (b["top"] + b["bottom"]) // 2
    
    @property
    def is_interactive(self) -> bool:
        """Determine if this element is likely interactive."""
        if self.clickable or self.checkable:
            return True
        interactive_classes = [
            "android.widget.Button", "android.widget.ImageButton",
            "android.widget.EditText", "android.widget.CheckBox",
            "android.widget.Switch", "android.widget.RadioButton",
            "android.widget.Spinner", "android.widget.ToggleButton",
        ]
        if self.class_name in interactive_classes:
            return True
        # Elements with content descriptions are often tappable
        if self.content_desc and self.enabled:
            return True
        return False
    
    def to_dict(self):
        return {
            "text": self.text,
            "resource_id": self.resource_id,
            "class": self.class_name,
            "content_desc": self.content_desc,
            "clickable": self.clickable,
            "checked": self.checked,
            "enabled": self.enabled,
            "bounds": self.bounds
        }


class UIHierarchy:
    """Parsed UI hierarchy for the current screen."""
    
    def __init__(self, xml_content):
        self.root = ET.fromstring(xml_content)
        self.elements = self._parse_all()
    
    def _parse_all(self):
        elements = []
        for node in self.root.iter("node"):
            elements.append(UIElement(node))
        return elements
    
    def get_interactive(self) -> list[UIElement]:
        """Get all interactive elements."""
        return [e for e in self.elements if e.is_interactive and e.enabled]
    
    def get_all(self) -> list[UIElement]:
        """Get all elements."""
        return self.elements
    
    def find_by_text(self, text: str, fuzzy: bool = False) -> UIElement | None:
        """Find element by text content."""
        for e in self.elements:
            if fuzzy:
                if text.lower() in e.text.lower():
                    return e
            else:
                if e.text == text:
                    return e
        return None
    
    def find_by_resource_id(self, resource_id: str) -> UIElement | None:
        """Find element by resource ID (exact or partial match)."""
        for e in self.elements:
            if resource_id in e.resource_id:
                return e
        return None
    
    def find_by_content_desc(self, desc: str, fuzzy: bool = False) -> UIElement | None:
        """Find element by content description."""
        for e in self.elements:
            if fuzzy:
                if desc.lower() in e.content_desc.lower():
                    return e
            else:
                if e.content_desc == desc:
                    return e
        return None
    
    @property
    def activity_name(self) -> str:
        """Get current activity from the hierarchy root."""
        # Activity is not in the hierarchy XML directly
        # Must be obtained separately via dumpsys
        return ""


def dump_hierarchy(device) -> UIHierarchy:
    """Dump and parse the current UI hierarchy."""
    device.shell("uiautomator dump /sdcard/apptest_ui.xml")
    device.pull("/sdcard/apptest_ui.xml", "/tmp/apptest_ui.xml")
    with open("/tmp/apptest_ui.xml") as f:
        xml_content = f.read()
    return UIHierarchy(xml_content)
```

### Element Finder (`element_finder.py`)

Merges UIAutomator hierarchy with OCR results to get complete element coverage.

```python
"""
Merge UI hierarchy elements with OCR results to find all interactive
elements on screen, including those missing from the hierarchy.

This is critical for apps using custom views, Jetpack Compose, Flutter,
or any framework where UIAutomator doesn't capture all elements.
"""

class MergedElement:
    """An element found from either hierarchy or OCR."""
    def __init__(self, source, text, bounds, center, element_type="unknown",
                 resource_id="", clickable="unknown", checked=None, enabled=True):
        self.source = source  # "hierarchy" or "ocr"
        self.text = text
        self.bounds = bounds
        self.center = center
        self.element_type = element_type
        self.resource_id = resource_id
        self.clickable = clickable
        self.checked = checked
        self.enabled = enabled
    
    def serialize_for_llm(self, index: int) -> str:
        """Format this element for inclusion in an LLM prompt."""
        parts = [f"[{index}]"]
        parts.append(f"type={self.element_type}")
        if self.text:
            parts.append(f'text="{self.text}"')
        if self.resource_id:
            parts.append(f"id={self.resource_id}")
        if self.checked is not None:
            parts.append(f"checked={self.checked}")
        parts.append(f"source={self.source}")
        parts.append(f"center=({self.center[0]},{self.center[1]})")
        return " ".join(parts)


def merge_hierarchy_and_ocr(hierarchy: UIHierarchy, ocr_results: list) -> list[MergedElement]:
    """
    Combine hierarchy elements with OCR results.
    OCR fills gaps where the hierarchy is incomplete.
    """
    merged = []
    matched_ocr_indices = set()
    
    # First: add all hierarchy elements
    for elem in hierarchy.get_interactive():
        merged.append(MergedElement(
            source="hierarchy",
            text=elem.text or elem.content_desc,
            bounds=elem.bounds,
            center=elem.center(),
            element_type=elem.class_name.split(".")[-1] if elem.class_name else "unknown",
            resource_id=elem.resource_id,
            clickable=elem.clickable,
            checked=elem.checked if elem.checkable else None,
            enabled=elem.enabled
        ))
        
        # Match against OCR results to avoid duplicates
        for i, ocr in enumerate(ocr_results):
            if i in matched_ocr_indices:
                continue
            if _texts_match(elem.text, ocr["text"]) and _bounds_overlap(elem.bounds, ocr["bounds"]):
                matched_ocr_indices.add(i)
    
    # Second: add OCR results that hierarchy missed
    for i, ocr in enumerate(ocr_results):
        if i not in matched_ocr_indices:
            bounds = ocr["bounds"]
            center = (
                (bounds["left"] + bounds["right"]) // 2,
                (bounds["top"] + bounds["bottom"]) // 2
            )
            merged.append(MergedElement(
                source="ocr",
                text=ocr["text"],
                bounds=bounds,
                center=center
            ))
    
    return merged


def serialize_elements_for_llm(elements: list[MergedElement]) -> str:
    """Serialize all elements into a string for LLM prompts."""
    lines = []
    for i, elem in enumerate(elements):
        lines.append(elem.serialize_for_llm(i))
    return "\n".join(lines)


def _texts_match(text1: str, text2: str) -> bool:
    """Fuzzy text match between hierarchy and OCR text."""
    if not text1 or not text2:
        return False
    t1 = text1.lower().strip()
    t2 = text2.lower().strip()
    return t1 == t2 or t1 in t2 or t2 in t1


def _bounds_overlap(bounds1: dict, bounds2: dict) -> bool:
    """Check if two bounding boxes overlap significantly."""
    if not bounds1 or not bounds2:
        return False
    x_overlap = max(0, min(bounds1["right"], bounds2["right"]) - max(bounds1["left"], bounds2["left"]))
    y_overlap = max(0, min(bounds1["bottom"], bounds2["bottom"]) - max(bounds1["top"], bounds2["top"]))
    overlap_area = x_overlap * y_overlap
    area1 = (bounds1["right"] - bounds1["left"]) * (bounds1["bottom"] - bounds1["top"])
    if area1 == 0:
        return False
    return overlap_area / area1 > 0.5
```

### Success Criteria Checking

Check whether a goal has been achieved using deterministic methods first, then LLM fallback:

```python
SUCCESS_CHECK_PROMPT = """
You are verifying whether a test goal has been achieved.

Goal: "{goal_description}"
Success criteria: "{success_criteria}"

Current screen:
- Activity: {current_activity}

Elements on screen:
{serialized_elements}

Based on what is currently on screen, is the success criteria met?

Respond as JSON:
{{"met": true|false, "confidence": "high|medium|low", "reasoning": "brief explanation"}}
"""

def check_success(goal, elements, hierarchy, device):
    """
    Check if goal success criteria are met.
    Try deterministic checks first, fall back to LLM.
    """
    criteria = goal["success_criteria"]
    
    # Deterministic check: screen/activity name match
    current_activity = get_current_activity(device)
    if "screen is visible" in criteria.lower():
        screen_name = criteria.lower().split("screen is visible")[0].strip()
        if screen_name in current_activity.lower():
            return True
    
    # Deterministic check: element text presence
    if "verify" in criteria.lower() and "appears" in criteria.lower():
        # Extract what should appear
        for elem in elements:
            if any(keyword in elem.text.lower() for keyword in extract_keywords(criteria)):
                return True
    
    # Deterministic check: toggle/checkbox state
    if "toggle" in criteria.lower() and "on" in criteria.lower():
        for elem in elements:
            if elem.checked is True and "switch" in elem.element_type.lower():
                return True
    
    # Fall back to LLM for complex criteria
    prompt = SUCCESS_CHECK_PROMPT.format(
        goal_description=goal["description"],
        success_criteria=criteria,
        current_activity=current_activity,
        serialized_elements=serialize_elements_for_llm(elements)
    )
    response = call_llm(prompt)
    result = parse_json(response)
    return result.get("met", False)
```

```python
        return None
    
    def summarize(self) -> str:
        """Produce a text summary for inclusion in LLM prompts."""
        lines = []
        for screen_id, info in self.screens.items():
            lines.append(f"Screen: {info['description']} (activity: {info['activity']})")
            for elem in info['key_elements']:
                lines.append(f"  - {elem}")
        
        for pattern in self.patterns:
            lines.append(f"Pattern: {pattern}")
        
        return "\n".join(lines) if lines else "No prior knowledge about this app."
    
    def save(self):
        """Persist knowledge base to disk."""
        os.makedirs(self.storage_path, exist_ok=True)
        data = {
            "app_package": self.app_package,
            "screens": self.screens,
            "actions": self.actions,
            "patterns": self.patterns
        }
        path = os.path.join(self.storage_path, f"{self.app_package}.json")
        with open(path, "w") as f:
            json.dump(data, f, indent=2)
    
    @classmethod
    def load(cls, app_package: str, storage_path: str = ".apptest/knowledge"):
        """Load knowledge base from disk, or create empty one."""
        kb = cls(app_package, storage_path)
        path = os.path.join(storage_path, f"{app_package}.json")
        if os.path.exists(path):
            with open(path) as f:
                data = json.load(f)
            kb.screens = data.get("screens", {})
            kb.actions = data.get("actions", [])
            kb.patterns = data.get("patterns", [])
        return kb
```

## Deterministic Screen Identification

Identify the current screen using deterministic signals from the UI hierarchy, not vision models.

```python
"""
Deterministic screen state identification.
Produces a stable hash from UI hierarchy signals.
No vision models, no LLM calls.
"""

import hashlib

def identify_screen(device, hierarchy: UIHierarchy) -> str:
    """
    Produce a deterministic screen identifier from hierarchy signals.
    Same screen state -> same ID, every time.
    """
    activity = get_current_activity(device)
    
    # Element type multiset (quantized to reduce sensitivity)
    type_counts = {}
    for elem in hierarchy.get_all():
        short_type = elem.class_name.split(".")[-1] if elem.class_name else "unknown"
        type_counts[short_type] = type_counts.get(short_type, 0) + 1
    
    # Quantize counts: 0, 1, few (2-5), many (6+)
    quantized = {}
    for t, count in sorted(type_counts.items()):
        if count == 0:
            q = "0"
        elif count == 1:
            q = "1"
        elif count <= 5:
            q = "few"
        else:
            q = "many"
        quantized[t] = q
    
    # Extract structural template: collapse repeated siblings
    template = extract_template(hierarchy)
    
    # Key identifiers
    nav_bar_title = ""
    for elem in hierarchy.get_all():
        if "toolbar" in elem.class_name.lower() or "actionbar" in elem.class_name.lower():
            if elem.text:
                nav_bar_title = elem.text
                break
    
    # Combine signals
    fingerprint = {
        "activity": activity,
        "nav_title": nav_bar_title,
        "type_distribution": quantized,
        "template": template,
        "has_keyboard": is_keyboard_shown(device)
    }
    
    # Hash to stable ID
    fingerprint_str = json.dumps(fingerprint, sort_keys=True)
    return hashlib.sha256(fingerprint_str.encode()).hexdigest()[:16]


def extract_template(hierarchy: UIHierarchy) -> str:
    """
    Extract a structural template from the hierarchy.
    Collapses repeated siblings so that a list with 5 items
    and a list with 50 items produce the same template.
    """
    # Simplified: use the element type tree with repeated children collapsed
    # Full implementation would walk the XML tree recursively
    types_in_order = []
    prev_type = None
    repeat_count = 0
    
    for elem in hierarchy.get_all():
        short_type = elem.class_name.split(".")[-1] if elem.class_name else "?"
        if short_type == prev_type:
            repeat_count += 1
        else:
            if repeat_count > 1:
                types_in_order.append(f"{prev_type}*N")
            elif prev_type:
                types_in_order.append(prev_type)
            prev_type = short_type
            repeat_count = 1
    
    # Don't forget the last element
    if repeat_count > 1:
        types_in_order.append(f"{prev_type}*N")
    elif prev_type:
        types_in_order.append(prev_type)
    
    return "|".join(types_in_order[:30])  # truncate for sanity
```

## Key Design Decisions

1. **CLI-first, not plugin-first.** A CLI runs in any CI system. Platform-specific plugins multiply engineering work. The customer writes 3-5 lines of CI config to integrate.

2. **Hierarchy + OCR, not vision models for element finding.** UIAutomator hierarchy is deterministic but incomplete. OCR is deterministic and catches what hierarchy misses. Together they provide complete element coverage without non-deterministic vision model calls.

3. **Goal-level planning, not action-level planning.** The LLM planner produces goals ("log in"), not actions ("tap the email field"). Action decisions happen at execution time with the real screen in front of the executor.

4. **Knowledge base as facts, not graph.** Store natural language facts about screens and actions. The LLM reasons over them. This is resilient to UI changes — if a button moves or gets renamed, the LLM adapts. A rigid graph would break.

5. **Deterministic screen identification.** Screen state is identified by hashing hierarchy signals (activity name, element type distribution, structural template). No vision models in the identification loop.

6. **LLM calls are minimized at execution time.** The knowledge base handles known screens and actions. LLM is called only for genuinely unknown situations. Over time, most execution becomes deterministic replay.

7. **Source code stays in customer's infra.** The CLI reads source locally, extracts only relevant context, and sends snippets to the LLM API. The full repo never leaves their environment.

8. **Logic changes are first-class citizens.** Most PRs change business logic, not UI layouts. The analyzer classifies every changed file, traces logic files through the dependency chain to their UI screen, and reads the actual code diff. The LLM generates tests by understanding code behavior, not just visual changes.

9. **Change nature drives test type.** The analyzer classifies changes as new features, bug fixes, refactors, error handling improvements, etc. The generator uses this to produce the right kind of tests — regression tests for refactors, error path tests for error handling changes, catch-the-bug tests for bug fixes.

**Key evaluation for logic changes specifically:**

- For an error handling change, did it test both the error path AND the happy path?
- Did it correctly translate code-level behavior into user-facing test steps?

**Run evaluation 3 times per PR** to measure consistency. If results vary wildly, constrain the prompt further.
